### 2. Architecture Principles

### 3. Project Structure Principles
- **Follow modern front-end best practices**: modular, shallow, predictable structure.
- Organize by **features/pages** with clear separation of concerns.
- **Pages**: keep `/pages` flat — each page as a single file (e.g., `Landing.tsx`, `Result.tsx`, `Error.tsx`), not separate folders.
- **Components**: keep `/components` flat — one file per shared UI primitive (`Button.tsx`, `InputField.tsx`, `Spinner.tsx`, etc.), no folder-per-component.
- **Features**: group related domain logic together under a single `/features/websiteCheck/` folder (e.g., input validation, result mapping, CTA handling), instead of splitting into multiple feature folders.
- **Utilities/Services**: place parsing, validation, lookup, and helpers in `/utils` or `/services`, separate from UI.
- **UI consistency**: shared primitives must only exist once in `/components` and be reused across all pages.
- **Keep it flat**: no folder-per-page or folder-per-component; only introduce subfolders for genuinely composite widgets (e.g., Modal with subparts).
- **Tests**: keep `/tests` top-level mirroring `/src`; do not over-nest.
- Produce a **high-level project folder tree** in a code block that reflects these rules: flat where possible, grouped only where it improves clarity.

### 4. Tech Stack Guidelines
- Use a **NextJS** with **TailwindCSS** for styling.
- Only introduce external libraries that address **real app needs** (e.g., URL parsing, schema validation).
- **Styling must be consistent** — enforce spacing, color, and typography tokens. Use utility-first CSS or modular CSS with design tokens for consistency.

### 5. Error Handling & Debugging Principles
- Clearly categorize errors: input validation, “not found,” network/timeouts.
- Always show **clear, actionable error messages** .
- Ensure recovery paths exist (retry, clear input, navigate back).

### 6. Deployment Principles
- **Build once → deploy bundle**: output must be deployable to Vercel.
